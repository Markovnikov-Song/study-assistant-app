# 项目中涉及应用人工智能大模型的内容说明

## （1）AI 使用说明

使用模型/工具：系统底层通过 FastAPI 的适配层动态接入主流大语言模型（LLM）的 API，支持 OpenAI 通用的 GPT 系列以及兼容 OpenAI 格式的国产模型（如硅基流动等）。

应用环节：

- **智能问答与 RAG 检索**：系统实现了完整的检索增强生成（RAG）流水线，在 `backend/services/rag_pipeline.py` 中实现。RAG Pipeline 支持三种工作模式：
  - **Strict 严格模式**：余弦相似度阈值过滤（默认 0.3），仅当检索到的文档块相关性满足阈值时才生成回答，从源头杜绝大模型幻觉
  - **Broad 混合模式**：结合本地向量检索与大模型内部知识，在确保标注来源的前提下补充通用知识
  - **Solve 解题模式**：结构化解题引导，按"考点→思路→步骤→踩分点→易错点"格式输出

- **方法论驱动（Skill 层）**：在 `lib/core/skill/` 目录下实现了完整的 Skill 执行引擎，允许用户配置特定的 Prompt 模板执行链（如费曼技巧、康奈尔笔记法），由 AI 扮演虚拟提问者或总结者引导用户深度思考

- **多智能体议事会（Agent Council）**：在 `lib/core/agent/` 和 `backend/routers/council.py` 中构建了多智能体协作治理机制，定义了校长（Principal）、班主任（Advisor）、老师（Teacher）、同桌（Companion）四种角色，通过甲状腺轴模型（快/中/慢反馈）实现自适应学习调控

- **学术文档自动化导出**：在 `backend/book_services/` 下实现了 Word 和 PDF 两种导出器，其中 `docx_book_exporter.py` 支持 LaTeX 公式渲染、目录生成、多种 block 类型排版

- **内容结构化**：AI 生成的结构化 Markdown 可一键解析为可视化导图节点

## （2）关键提示词与输出样例

| 场景模块 | 系统核心 Prompt 逻辑 | AI 输出响应样例（摘要） |

# 第一章 需求分析

## 1.1 项目背景与政策契合度

随着国家"人工智能+"行动计划的深入推进，教育领域正经历从"信息化"向"智能化"的数字化转型。2024年教育部明确提出要构建"数字化驱动的教育教学新生态"，鼓励利用大模型技术赋能学生个性化学习。然而，当前的数字化教学资源虽丰富，却存在"资源碎片化、工具散乱化、方法论缺失"的问题。全学科学生在学习中均面临知识密度大、公式/逻辑推导复杂、文档专业性强的痛点，传统的通用协作软件难以满足深度学习的需求。本项目《伴学 - AI 智能学习助手》正是响应教育数字化政策，旨在通过 RAG（检索增强生成）与多智能体（Multi-Agent）技术，为全体学生构建一个标准化的、方法论驱动的智能学习空间。

## 1.2 目标用户分析与痛点识别

本项目主要面向高等院校本科生及研究生、中学生等全学段学生，覆盖文、理、工、医、经、管等全学科，核心痛点如下：

1. **知识"幻觉"风险**：通用 AI 模型在回答专业细节时易产生幻觉，学生缺乏可信赖的辅助工具
2. **认知负荷过重**：学习过程需频繁切换 PDF 阅读器、笔记软件、日历排期和搜索引擎，导致注意力频繁中断
3. **学习方法论落地难**：费曼技巧、康奈尔笔记、间隔重复等优秀方法论在传统软件中需人工维护，学生难以坚持
4. **缺乏自适应反馈**：现有的日程表是静态的，无法根据用户的错题反馈、学习速率波动进行动态的计划调整

## 1.3 核心功能需求

基于以上痛点，系统必须满足以下四大核心功能：

| 功能编号 | 功能名称 | 功能描述 | 代码实现位置 |
|---------|---------|---------|-------------|
| F1 | 全学科 RAG 知识库 | 支持用户上传各学科教材、课件及论文，系统实现精准的文档切片与语义检索，具备"Strict 严格模式" | `backend/services/rag_pipeline.py` |
| F2 | Skill 驱动的学习执行引擎 | 将抽象的学习方法抽象为可调度的 Prompt 链，由 AI 引导用户一步步完成深度思考 | `lib/core/skill/skill_model.dart`, `lib/core/skill/skill_io_service.dart` |
| F3 | Multi-Agent 议事会调度系统 | 分配校长、班主任、老师、同桌四种智能体角色，实现"学、测、调"闭环 | `lib/core/agent/agent_council.dart`, `backend/routers/council.py` |
| F4 | 原生可视化知识生产 | 提供思维导图与学术级 .docx/.pdf 讲义导出功能，支持 LaTeX 公式渲染 | `lib/components/mindmap/`, `backend/book_services/` |

## 1.4 非功能性需求

- **数据隔离与安全性**：确保多学科会话状态完全隔离，用户私有学习资料在向量化过程中的数据隐私
- **多端响应能力**：系统基于 Flutter 适配手机、平板及 Web/PC 等不同交互场景
- **响应性能**：大模型交互采用 SSE 流式响应，前端实现逐字打字机效果，提升心流学习体验
- **向量检索性能**：使用 PGVector 向量数据库，支持高效的余弦相似度检索

## 1.5 竞品对比与差异化价值

| 竞品 | 优势 | 不足 | 伴学解决方案 |
|------|------|------|-------------|
| Cognito/Anki | 单点记忆功能完善 | 缺乏对整本教材的理解能力，无跨学科规划 | 内置 RAG 知识库，支持学科上下文理解 |
| MindPal/Notion AI | 企业协作与工作流成熟 | 学习方法论垂直深度不足，学术公式支持差 | Skill 引擎深度集成学习方法论，LaTeX 完整支持 |
| 伴学 (Learning OS) | 工具+方法+陪伴三位一体 | — | 首次提出"学习专用操作系统"概念，实现从"AI 提供答案"到"AI 辅助思考"的本质跨越 |

# 第二章 概要设计

## 2.1 设计理念与架构范式

本项目摒弃了传统学习软件"功能堆砌"的模式，创新性地提出了 **"Learning OS（学习专用操作系统）"** 架构范式。核心理念：**工具归工具，方法归方法，用户说了算**。核心逻辑在于将底层原子工具能力（如记录、渲染、检索）与高层学习方法论调度（如 Skill、Plan、Agent）进行深度解耦。系统被设计为一个微内核（Micro-kernel）架构，其中 AI 智能体充当操作系统的调度器，而各个学习组件（笔记本、思维导图、题库）则作为标准化的系统服务被调用。

## 2.2 系统整体架构图

系统采用严密的"前后端分离"与"分层依赖"模型，分为以下五层结构：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        用户感知层 (Client UI)                        │
│    Flutter 跨平台框架 + Riverpod 状态管理 + GoRouter 路由导航         │
│    适配 Android/iOS/Web/PC，组件化设计，响应式布局                     │
└─────────────────────────────────────┬───────────────────────────────┘
                                      │
┌─────────────────────────────────────┴───────────────────────────────┐
│                        模式控制层 (Control Layer)                      │
│    四种运行模式：纯手动模式 / DIY 模式 / Skill 驱动模式 / Agent 陪伴模式 │
└─────────────────────────────────────┬───────────────────────────────┘
                                      │
┌─────────────────────────────────────┴───────────────────────────────┐
│                        业务组件层 (Component Ecosystem)                │
│    答疑室 / 笔记本 / 思维导图 / 错题本 / 资料库 / 排期日历              │
│    组件间通过 EventBus 解耦通信，统一接口协议                          │
└─────────────────────────────────────┬───────────────────────────────┘
                                      │
┌─────────────────────────────────────┴───────────────────────────────┐
│                        原子工具层 (Tooling Utilities)                  │
│    Canvas 绘图 / Docx 底层 XML 解析器 / LaTeX 公式渲染 / SSE 客户端    │
│    XMind/FreeMind 导入解析 / 富文本编辑器 (flutter_quill)             │
└─────────────────────────────────────┬───────────────────────────────┘
                                      │
┌─────────────────────────────────────┴───────────────────────────────┐
│                        核心后端与数据服务层 (FastAPI Backend)          │
│    FastAPI 异步框架 + SQLite/Neon PostgreSQL + PGVector 向量数据库     │
│    RAG Pipeline / LLM Service / Agent Council / Skill Ecosystem      │
└─────────────────────────────────────────────────────────────────────┘
```

## 2.3 多智能体议事会 (Multi-Agent Council) 模型

为实现"陪伴式学习"并解决单一智能体能力边界问题，系统构建了多智能体协作治理模型：

| 智能体角色 | 职责定义 | 反馈响应类型 | 代码实现 |
|-----------|---------|-------------|---------|
| 校长 (Principal) | 负责月/季度的长期学术目标规划，调控宏观复习进度 | 慢反馈 (Slow Signal) | `lib/core/agent/agent_role.dart` |
| 班主任 (Advisor) | 核心排期调度者，通过日历 API 插入或优化每日学习 Task | 中反馈 (Medium Signal) | `backend/routers/council.py` |
| 各科老师 (Teacher) | 全学科知识库调用者，负责执行具体的 Skill 脚本 | 任务驱动 (Task Driven) | `lib/core/skill/` |
| 同桌 (Companion) | 实时监控者，监听用户心流、准确率及压力数据，触发即时干预 | 快反馈 (Fast Signal) | `lib/core/agent/agent_council_impl.dart` |

## 2.4 数据流向与状态管理

**响应式状态流**：客户端采用 Riverpod 2.6 进行状态管理。利用 Family Provider 实现全学科、多会话的物理隔离，确保用户在切换"高等数学"与"大学英语"会话时，内存数据不发生交叉感染。

**RAG 数据流**：`backend/services/rag_pipeline.py`
```
原始文档（PDF/Docx）→ 后端分词切块 → OpenAI Embedding 向量化 → PGVector 存储 
→ 问答时计算余弦相似度检索上下文 → LLM 生成回复 → SSE 流式推送前端
```

**反馈控制流**：`lib/core/agent/agent_council.dart`
```
用户错题 → 同桌 Agent 感知 → 计算负反馈强度（快/中/慢）→ 班主任/校长 Agent 决策
→ 日历数据修改 → 前端 UI 实时重绘日程表
```

## 2.5 技术栈选型理由

| 层级 | 技术 | 版本 | 选型理由 | 代码佐证 |
|------|------|------|---------|---------|
| 前端框架 | Flutter | ≥3.11.4 | 高性能绘图能力支撑思维导图，多端一致性减少开发成本 | `pubspec.yaml` |
| 状态管理 | Riverpod | 2.6 | 编译时安全，Family Provider 实现会话隔离 | `lib/providers/` |
| 路由 | GoRouter | 14.6 | 声明式路由，支持深层链接和路径参数 | `lib/routes/app_router.dart` |
| HTTP 客户端 | Dio | 5.7 | 拦截器统一处理认证、错误、日志 | `lib/core/network/dio_client.dart` |
| 后端框架 | FastAPI | 0.95+ | 异步特性完美契合大模型流式响应 SSE | `backend/main.py` |
| 数据库 | SQLite + PGVector | — | 轻量级本地部署，PGVector 支持向量语义检索 | `backend/database.py`, `backend/services/rag_pipeline.py` |
| 前端 SSE | 自研 SSE Client | — | 跨平台统一实现 (Native/Web) | `lib/tools/network/sse_client_*.dart` |

# 第三章 详细设计

## 3.1 智能问答与分层 RAG 检索管线

系统在 `backend/services/rag_pipeline.py` 中实现了完整的检索增强生成管线，适配全学科文档（教材、课件、笔记、论文），设计细节如下：

### 3.1.1 分级检索策略

**语义预处理**：利用 langchain 的 PDF 解析器对上传的 PDF 进行层级解析，保留章节标题作为元数据，防止语义分片断裂。

**多模式路由**：

| 模式 | 描述 | 相似度阈值 | 使用场景 |
|------|------|-----------|---------|
| Strict（严格模式） | 仅使用本地资料回答，未匹配时返回"查无依据" | τ=0.3（距离值越小越相关） | 高严谨度学科（理工医法） |
| Broad（混合模式） | 本地资料+通用知识混合，标注来源 | 无阈值 | 文史经管等解释性概念 |
| Solve（解题模式） | 结构化解题过程输出 | 无阈值 | 数学/物理等解题辅导 |

**SSE 流式响应协议**：`backend/services/llm_service.py` + `lib/tools/network/sse_client_*.dart`
- 后端 FastAPI 通过 `StreamingResponse` 实时推送 Token
- 前端 SSE 客户端跨平台统一实现（`sse_client_native.dart` / `sse_client_web.dart`）
- 实现逐字打字机效果，提升交互体验

## 3.2 多智能体议事会 (Agent Council) 运行机制

本项目核心创新点在于 `lib/core/agent/` 下定义的议事会机制，代码结构如下：

| 模块 | 文件 | 职责 |
|------|------|------|
| 接口定义 | `agent_council.dart` | 定义 AgentRegistry、CouncilAgenda、CouncilDecision 等核心接口 |
| 角色定义 | `agent_role.dart` | 定义 AgentRoleType 枚举（Principal/Advisor/Teacher/Companion） |
| 提示词模板 | `agent_prompt_template.dart` | 各角色的 Prompt 模板 |
| 核型调度 | `agent_kernel.dart` / `agent_kernel_impl.dart` | 单 dispatcher 向 AgentCouncil 的过渡兼容层 |
| 后端实现 | `backend/routers/council.py` | FastAPI 路由处理 Agent 协作请求 |

**负反馈回路算法**：
```
挫败感因子 = 错题数量 × 难度权重 × 时间衰减因子
当 挫败感因子 > 阈值：
  → 快反馈：立即调整学习方式（如切换为引导模式）
  → 中反馈：每日调整计划（班主任 Agent 插入补课任务）
  → 慢反馈：每周调整策略（校长 Agent 重排复习大纲）
```

## 3.3 思维导图渲染引擎 (MindMap Engine)

系统采用 **WebView + Markmap** 方案实现思维导图渲染，而非自研 Canvas 引擎：

**核心实现** (`lib/components/mindmap/`)：
- `widgets/mindmap_editor_canvas.dart`：WebView 容器组件
- `providers/mindmap_providers.dart`：Riverpod 状态管理
- `tools/import_parser.dart`：支持导入 XMind (.xmind) 和 FreeMind (.mm) 格式

**Markdown ↔ JSON ↔ NodeTree 三向转换**：
- AI 生成的结构化 Markdown 可一键解析为可视化导图节点
- 支持导出为图片（使用 `screenshot` 包）

## 3.4 学术级文档自动化导出技术

系统在 `backend/book_services/` 下实现了两种导出器：

### 3.4.1 Word 导出器 (`docx_book_exporter.py`)

- **目录生成**：`_build_toc()` 方法生成带书签超链接的目录
- **LaTeX 公式渲染**：`LatexRenderer` 类将 LaTeX 转为 PNG 内嵌到 Word
- **Block 类型支持**：heading / paragraph / code / list / quote
- **样式控制**：支持中文字体（宋体）、代码字体（Courier New）、自定义配色

### 3.4.2 PDF 导出器 (`pdf_book_exporter.py`)

使用 `reportlab` 库实现，支持与 Docx 导出相同的 block 类型和样式。
